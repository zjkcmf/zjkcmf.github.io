TCP知识点备忘

发表于 2019-11-20

0.TCP的特点

  • 可靠连接
  • 字节流服务

1.建立连接的过程

  • A -> B (syn, seq=0, ack=0, A:sync_sent, B:)
  • B -> A (syn, seq=0, ack=1, A:esdablish, B:)
  • A -> B (seq=1, ack=1, A:, B:esdablish)

2.断开连接的过程

  • A -> B (fin, A:fin_wait1)
  • B -> A (ack, A:fin_wait2)
  • B -> A (fin, A:time_wait)
  • A -> B (ack, A:close)

3.为什么需要3次握手和4次挥手？

  • 3次握手保证了客户端、服务器双方都确认收到了对方的ack，且次数最小
  • 4次挥手是因为TCP为全双工，前两次fin和ack关闭了一个方向的连接，另一个方向不影响

4.协议头详解

  • 协议头最多60个字节
  • 除选项外20个字节
  • 一个包可以不携带数据
  • 包含源端口和目标端口
  • 包含标志位，如syn, fin, ack等等
  • 包含seq序列号和确认序列号，如果某一端连续发送多个包，ack是一样的
  • 序列号为已经发送的包大小加一
  • 确认序列号为已经接受的包大小加一
  • 包含紧急指针

5.wireshark抓包分析

如下展示的是本机的postman向jenkins服务器发起的http请求过程

  • 1.客户端发起syn，图中从ff开始到00结束的为TCP的头部，共20个字节（2个16进制为一个字节）
  • 2.服务器响应syn|ack，ack为1
  • 3.客户端响应ack
  • 4.客户端发送HTTP请求
  • 5.服务端ack=282，为4的客户端包序号加上payload大小
  • 6.服务端向客户端发送数据，包含从210开始到215的共6个数据包。因为包太大，做了拆分
  • 7.客户端向服务端发送ack，其中第一个ack包含了前两个数据包的回应
  • 8.客户端因收了大量数据，占用缓存区，调整窗口大小
  • 9.客户端发送keepalive包给服务端（TODO: 找到原因）,keepalive包的标志位为ack，seq为上一个seq-1，ack为上一个ack
  • 10.服务端发起fin
  • 11.客户端ack
  • 12.客户端发起fin，并回复ack
  • 13.服务端ack,结束断开

6.keepalive

  • 目的是为了在复杂的网络环境下保证tcp连接的存活
  • 一般发送心跳包的空闲时间间隔为7200秒，也就是两小时
  • 心跳包的seq和ack为上一次的值

7.流控机制

  • 连接建立后，对方会发来窗口调整包，指明窗口大小
  • 窗口大小代表了对方能一次接受多少包才给ack

8.Nagle算法和延迟ack

  • Nagle算法可以理解为：只有收到对方的ack，才能发送数据，在等待ack过程中要发送的数据包都攒到一起发送
  • 延迟ack：为了避免ack包单独发送，会延迟一定时间，如果正好有需要发送的数据，就捎带一起发送过去
  • 两者结合，可能出现的问题：发送端由于没收到ack，一直等待，接受段等待ack延迟时间，不发送ack
